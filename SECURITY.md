# Security notes — Spaces Coworking

## The one thing to understand first

The staff app talks to Supabase **directly from the browser**. Anything this
repository does to slow an attacker down can be skipped by simply not loading
our page and posting straight to the Supabase auth endpoint.

So the login throttle in `app.js` is a **convenience control**, not a defence.
It stops a person guessing at the keyboard and it stops a script that drives
our page. It does not stop a determined attacker. **The controls that actually
stop one live in the Supabase dashboard, and only the owner can turn them on.**

## Measured on 2026-07-21

A probe sent 26 consecutive failed logins to the production auth endpoint.
All 26 returned `400 invalid_credentials`. No lockout, no rate limit, no
CAPTCHA challenge. Password guessing is currently limited only by network
speed.

One thing already correct: the error is a generic `Invalid login credentials`
whether or not the email exists, so accounts cannot be enumerated. The app
keeps that property by never echoing the server's wording back to the user.

## Required — do these in the Supabase dashboard

1. **Turn on CAPTCHA (highest value).**
   Authentication → Settings → Bot and Abuse Protection → enable hCaptcha or
   Cloudflare Turnstile. This is the single change that ends automated password
   guessing. It requires a matching change in `login()` to pass the token, so
   tell me once it is enabled and I will wire it up.

2. **Lower the auth rate limits.**
   Authentication → Rate Limits. Reduce sign-in attempts per hour per IP to
   something small (10–30). This is server-side and cannot be bypassed.

3. **Require strong passwords.**
   Authentication → Settings → minimum length 12 and require mixed character
   types. Every staff password should be unique and stored in a password
   manager.

4. **Enable leaked-password protection.**
   Authentication → Settings. Rejects passwords found in known breaches.

5. **Enable MFA for the owner account.**
   The owner login can read every payment, ledger and audit row. It deserves a
   second factor.

6. **Rotate the service-role key if it has ever been shared.**
   Settings → API. That key bypasses every row-level security rule. It belongs
   only in Vercel environment variables, never in the repository, never in a
   chat message, never in client JavaScript.

7. **Turn on database backups / PITR.**
   This is the financial system of record. Database → Backups.

## What the code already does

- **Progressive login lockout** (`app.js`): after 2 failed attempts the form
  locks for 5s, then 15s, 60s and 300s, capped at 5 minutes. Stored in
  `localStorage`, so a page refresh does not clear it. A network failure never
  counts as a failed credential, and a successful login resets the counter.
- **No account enumeration**: the message is always "Email or password is
  incorrect", regardless of which was wrong.
- **The password field is cleared** after every attempt, successful or not.
- **No staff-login link on the public site.** `/admin` still exists and is not
  a secret, but it is not advertised, and it carries `noindex, nofollow`.
- **Admin response headers** (`vercel.json`): `X-Frame-Options: DENY` and
  `frame-ancestors 'none'` prevent clickjacking, `Referrer-Policy: no-referrer`
  keeps the admin URL out of other sites' logs, and `Cache-Control: no-store`
  keeps admin pages off shared disks.
- **Row-level security** governs every table. Staff cannot read payments, the
  owner ledger, the audit log or website analytics; those are owner-only. Rows
  generated by a receipt are locked so the books cannot drift from the invoices.
- **Money is written only through `SECURITY DEFINER` functions** that check for
  an active staff account first, so a stolen anon key alone cannot write an
  invoice.

## What is safe to be public

`supabaseConfig.anonKey` in `app.js` is *designed* to be public. It grants
nothing on its own — row-level security decides what any request may do. Do not
confuse it with the service-role key, which grants everything and must never
appear in client code.
